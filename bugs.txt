### Vấn đề 1: Bỏ sót bộ lọc RAM (`memoryPendingReadIds`) khi lấy tin ngẫu nhiên từ Cache
**Nguyên nhân:**
Hàm `markNewsAsRead` thêm ID vào `memoryPendingReadIds` để chặn hiển thị lại ngay lập tức. Tuy nhiên, khi hàm `getRandomNewsFromCache()` gọi xuống DB để lấy ID chưa đọc (`getUnreadNewsIdsFromFeedsNoExclude`), nó **quên không loại trừ** các ID đang nằm trong `memoryPendingReadIds` này. Điều này khiến tin vừa đọc xong có thể xuất hiện lại ngay ở lần random tiếp theo.

**Cách Fix:** Sửa lại hàm `getRandomNewsFromCache` trong `RssRepository.kt` để lọc qua bộ nhớ RAM trước khi random.

```kotlin
// Trong RssRepository.kt
suspend fun getRandomNewsFromCache(count: Int = 5): List<NewsItem> = withContext(Dispatchers.IO) {
    val enabledFeeds = rssFeedDao.getEnabledFeedsList()
    if (enabledFeeds.isEmpty()) return@withContext emptyList()

    val feedIds = enabledFeeds.map { it.id }
    val excludedInSession = emptyList<String>() 
    
    // 1. Get ALL candidates IDs
    val candidateIds = if (excludedInSession.isEmpty()) {
        cachedNewsDao.getUnreadNewsIdsFromFeedsNoExclude(feedIds)
    } else {
        cachedNewsDao.getUnreadNewsIdsFromFeeds(feedIds, excludedInSession)
    }
    
    // [FIX CODE] Lọc bỏ các ID vừa đọc đang lưu tạm trên RAM
    val unreadCandidates = candidateIds.filter { it !in memoryPendingReadIds }

    if (unreadCandidates.isEmpty()) {
        DebugLogger.log("READ", "getRandomNewsFromCache: No unread candidates")
        return@withContext emptyList()
    }

    // 2. Shuffle in Memory 
    val selectedIds = unreadCandidates.shuffled().take(count)
    DebugLogger.log("READ", "getRandomNewsFromCache: Pool=${unreadCandidates.size}, Picking=${selectedIds.size}")
    
    // 3. Fetch full items
    val cached = cachedNewsDao.getNewsByIds(selectedIds)
    return@withContext cached.map { it.toNewsItem() }
}
```

### Vấn đề 2: Nguy cơ Crash ứng dụng do giới hạn biến SQLite (SQLite Variable Limit)
**Nguyên nhân:**
Khi tải tin từ mạng về (có thể lên tới hàng ngàn tin), hàm `filterOutReadNews` truyền toàn bộ danh sách ID vào truy vấn `IN (:newsIds)` của SQLite. Mặc định SQLite giới hạn tối đa 999 biến trong một câu lệnh, nếu vượt qua sẽ văng lỗi `SQLiteException: too many SQL variables` và gây crash app.

**Cách Fix:**
Cắt nhỏ (chunk) danh sách ID thành các mảng nhỏ (ví dụ: 900 phần tử) để truy vấn an toàn.

```kotlin
// Trong RssRepository.kt
private suspend fun filterOutReadNews(news: List<NewsItem>): List<NewsItem> {
    if (news.isEmpty()) return news
    val candidateIds = news.map { it.id }
    
    // [FIX CODE] Chunk danh sách ID để tránh giới hạn 999 biến của SQLite
    val readIds = mutableSetOf<String>()
    candidateIds.chunked(900).forEach { chunk ->
        readIds.addAll(readNewsDao.getAlreadyReadIds(chunk))
    }
    
    if (readIds.isEmpty() && memoryPendingReadIds.isEmpty()) return news
    
    return news.filter { it.id !in readIds && it.id !in memoryPendingReadIds }
}
```

### Vấn đề 3: UI bị lỗi Logic do quản lý trạng thái bằng Index thay vì ID
**Nguyên nhân:**
Trong `MainViewModel`, biến `readIndicesInSession` (chứa Index) được dùng để đánh dấu bài đã qua khi bấm Next/Prev. Tuy nhiên, nếu danh sách `_uiState.value.newsItems` bị làm mới (auto-refresh ngầm) thì các Index này không còn đúng nữa (Index 0 của list cũ khác Index 0 của list mới). Điều này dẫn tới lỗi bỏ qua tin chưa đọc hoặc lặp vô tận.

**Cách Fix:**
Thay thế việc lưu `Index` bằng việc lưu `ID` của bài báo trong `MainViewModel.kt`.

```kotlin
// Trong MainViewModel.kt
// [FIX CODE] Thay Set<Int> thành Set<String> để lưu theo ID thay vì Index
private val readIdsInSession = mutableSetOf<String>()

fun selectRandomOtherNews() {
    val now = System.currentTimeMillis()
    if (now - lastSelectTime < DEBOUNCE_MS) return
    lastSelectTime = now

    val newsItems = _uiState.value.newsItems
    if (newsItems.isEmpty()) return

    val currentSelectedNewsId = _uiState.value.newsItems.getOrNull(_selectedNewsIndex.value)?.id
    
    // [FIX CODE] Lọc ra các Index của bài báo có ID chưa nằm trong danh sách đã đọc của phiên
    val availableIndices = newsItems.indices.filter { index ->
        val item = newsItems[index]
        item.id != currentSelectedNewsId && item.id !in readIdsInSession
    }
    
    if (availableIndices.isEmpty()) {
        Log.d(TAG, "All items read in session, fetching new batch...")
        readIdsInSession.clear()
        refreshNews()
    } else {
        val randomIndex = availableIndices.random()
        val nextNewsItem = newsItems[randomIndex]
        readIdsInSession.add(nextNewsItem.id) // Lưu ID thay vì index
        selectNews(randomIndex)
    }
}

fun refreshNews(force: Boolean = false) {
    // ... code giữ nguyên ...
    if (force) {
        _selectedNewsIndex.value = -1
        readIdsInSession.clear() // Xóa cache ID của session
        _uiState.value = _uiState.value.copy(hasReadHistory = false)
    }
    // ...
}
```

### Vấn đề 4: Sai lệch nghiệp vụ giữa Comment và Logic DB ("Hôm nay" vs "Mãi mãi")
**Nguyên nhân:**
Document code ghi `Filter out news items that have been read today` (Chỉ lọc tin đã đọc hôm nay). Nhưng truy vấn `@Query("SELECT newsId FROM read_news WHERE newsId IN (:newsIds)")` không hề kiểm tra thời gian. Do App giữ lịch sử 7 ngày, nên thực tế app đang cấm hiển thị lại các tin đã đọc trong vòng 7 ngày qua.

**Cách Fix:**
Cập nhật SQL Query trong `ReadNewsDao` và truyền thêm `todayStart` từ Repository.

```kotlin
// 1. Sửa trong ReadNewsDao.kt
@Query("SELECT newsId FROM read_news WHERE newsId IN (:newsIds) AND readAt >= :todayStart")
suspend fun getAlreadyReadIds(newsIds: List<String>, todayStart: Long): List<String>

// 2. Cập nhật lại lời gọi hàm trong RssRepository.kt (hàm filterOutReadNews)
private suspend fun filterOutReadNews(news: List<NewsItem>): List<NewsItem> {
    if (news.isEmpty()) return news
    val candidateIds = news.map { it.id }
    
    val todayStart = getTodayStartTimestamp() // Lấy mốc 0h hôm nay
    
    val readIds = mutableSetOf<String>()
    candidateIds.chunked(900).forEach { chunk ->
        // [FIX CODE] Truyền thêm tham số todayStart vào Dao
        readIds.addAll(readNewsDao.getAlreadyReadIds(chunk, todayStart))
    }
    
    if (readIds.isEmpty() && memoryPendingReadIds.isEmpty()) return news
    return news.filter { it.id !in readIds && it.id !in memoryPendingReadIds }
}
```

### Vấn đề 5: Rò rỉ bộ nhớ (Memory Leak) với các `Set` RAM In-Memory
**Nguyên nhân:**
Hai tập hợp `memoryPendingReadIds` (trong RssRepository) và `markedReadInService` (trong MainViewModel) chỉ có thao tác thêm `.add()` mà không bao giờ bị xóa. Nếu người dùng mở app đọc tin liên tục nhiều ngày, RAM sẽ cạn kiệt dần.

**Cách Fix:**
Làm sạch các cấu trúc dữ liệu này khi người dùng chủ động làm mới ứng dụng (Force Refresh) hoặc định kỳ dọn dẹp.

```kotlin
// 1. Trong MainViewModel.kt - Cập nhật hàm refreshNews:
if (force) {
    _selectedNewsIndex.value = -1
    readIdsInSession.clear()
    markedReadInService.clear() // [FIX CODE] Dọn dẹp RAM rác Service
    _uiState.value = _uiState.value.copy(hasReadHistory = false)
}

// 2. Trong RssRepository.kt - Thêm hàm dọn dẹp RAM và gọi nó khi dọn dẹp DB
suspend fun cleanupOldReadHistory() = withContext(Dispatchers.IO) {
    val retentionDays = com.skul9x.rssreader.utils.AppConfig.READ_HISTORY_RETENTION_DAYS
    val cutoffTime = System.currentTimeMillis() - (retentionDays * 24 * 60 * 60 * 1000L)
    readNewsDao.deleteOlderThan(cutoffTime)
    
    // [FIX CODE] Xóa bớt bộ nhớ tạm trên RAM (để an toàn, ta clear hẳn hoặc xóa các ID đã vào DB)
    memoryPendingReadIds.clear() 
}
```

### Vấn đề 6: Race Condition khi Service đẩy sự kiện (Đồng bộ Coroutine)
**Nguyên nhân:**
Trong `observeServiceState()`, khi Service báo đang đọc Index mới, ViewModel sẽ lưu xuống DB bằng `launch(Dispatchers.IO)`. Do `Dispatchers.IO` chạy đa luồng đồng thời, nếu nhảy bài nhanh (Next/Prev spam), các lệnh ghi DB có thể thực thi lộn xộn, sai thứ tự thời gian, gây lỗi lịch sử đồng bộ Firebase.

**Cách Fix:**
Sử dụng một dispatcher tuần tự (`limitedParallelism(1)`) chuyên dùng cho việc ghi log/history để đảm bảo lệnh nào gọi trước sẽ được ghi DB trước.

```kotlin
// Trong MainViewModel.kt
// [FIX CODE] Tạo một dispatcher đơn luồng để chuyên xử lý việc ghi lịch sử tránh xung đột
private val historyWriteDispatcher = Dispatchers.IO.limitedParallelism(1)

private fun observeServiceState() {
    viewModelScope.launch {
        com.skul9x.rssreader.service.NewsReaderService.serviceState.collect { serviceState ->
            // ... code cập nhật UI ...
            
            if (serviceState.currentIndex >= 0) {
                _selectedNewsIndex.value = serviceState.currentIndex
                val currentList = serviceState.continuousNewsItems ?: _uiState.value.newsItems
                val currentNews = currentList.getOrNull(serviceState.currentIndex)
                
                if (currentNews != null && currentNews.id !in markedReadInService) {
                    markedReadInService.add(currentNews.id)
                    // [FIX CODE] Dùng historyWriteDispatcher để đảm bảo ghi tuần tự 1-1
                    launch(historyWriteDispatcher) {
                        repository.markNewsAsRead(currentNews.id)
                    }
                }
            }
        }
    }
}
```
